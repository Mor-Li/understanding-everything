.github/ISSUE_TEMPLATE/bug.yml

这份文件虽然看着像代码，但其实它并不是一段程序逻辑，而是一份“问卷调查表”的配置文件。

你可以把它想象成去医院看病时的“挂号单”，或者是去修电脑时的“故障报修单”。这份文件的作用是告诉 GitHub（代码托管平台）：“当有人来反馈 Bug（漏洞）时，请按我设定的格式让他填表。”

为了让你完全理解它的逻辑，我为你列了一个 “学习任务清单（Task List）”，我们一步步拆解这份“报修单”背后的观点和要求。

任务 1：理解它的身份（Header 部分）

对应代码：

name: 🐞 Bug Report
description: Create a report to help reproduce and resolve the bug.
title: "[Bug] "
labels: ["bug"]

解读观点： * 观点：分类管理很重要。 * 讲解：这部分定义了这张表单的名字叫“Bug Report”（Bug 报告）。 * 当你点击“新建问题”时，系统会自动给标题加上 [Bug] 前缀，并打上 bug 的标签。 * 目的：让开发者一眼就能看出这是个错误报告，而不是功能建议或闲聊。

任务 2：设置门槛（Checklist 部分）

对应代码：

  - type: checkboxes
    ...
    options:
      - label: "I have checked FAQs ... and existing issues..."
      - label: "Please report this bug in English..."

解读观点： * 观点：不要做伸手党，尽量用通用语言。 * 讲解：这是表单的第一关，强制用户打钩确认两件事： 1. 拒绝重复劳动：你提问前，必须先去查查 FAQ（常见问题）和以前的帖子，别拿已经解决过的问题来烦我。 2. 全球化沟通：请用英语提问，这样全世界的开发者都能看懂并帮你，也方便以后的人搜索。

任务 3：说清楚“挂了什么科”（Description 部分）

对应代码：

  - type: textarea
    id: description
    attributes:
      label: Describe the Bug
    validations:
      required: true

解读观点： * 观点：必须要有一个清晰的摘要。 * 讲解：这是一个必填的大文本框。 * 目的：让用户用文字简单描述发生了什么错误。比如：“我运行了 A 函数，结果报错退出了。”

任务 4：最核心的“呈堂证供”（Reproduction 部分）

对应代码：

  - type: textarea
    id: reproduction
    attributes:
      label: Steps to Reproduce the Bug
      description: |
        Please include a code sample...
        Avoid using screenshots...

解读观点： * 观点：无代码，不修图；截图是无用的。 * 讲解：这是整个文件最强硬的观点部分。 * “给我代码”：开发者要求用户提供一段可以运行的代码（Code Sample），这样开发者能在自己的电脑上重现这个错误。 * “别发截图”：代码明确写了 Avoid using screenshots。因为截图里的代码没法复制粘贴，开发者不想照着图片手敲代码来测试。 * 潜台词：如果你不能提供复现步骤，我就没法修，这体现了“可复现性”在工程中的重要地位。

任务 5：心理落差对比（Expected Behavior 部分）

对应代码：

  - type: textarea
    id: expected-behavior
    attributes:
      label: Expected Behavior

解读观点： * 观点：逻辑错误比报错更难搞，所以要对比。 * 讲解： * 有时候程序没报错，但算出的结果是错的（比如 1+1 算成了 3）。 * 这一步要求用户写下：“你原本期望发生什么？” * 通过对比“实际发生的”和“期望发生的”，开发者能更快定位逻辑漏洞。

任务 6：案发现场调查（Environment Info 部分）

对应代码：

  - type: textarea
    id: environment-info
    attributes:
      label: Environment Information
      value: |
        1. Torch:
        2. Triton:

解读观点： * 观点：软件版本不兼容是万恶之源。 * 讲解： * 很多 Bug 是因为用户用的软件版本太旧或太新导致的。 * 这里预设了格式，强行要求用户填上 Torch 和 Triton（这是两个深度学习相关的库）的版本号。 * 目的：排除“环境配置错误”这种低级原因。

总结：这个文件的核心逻辑

如果把这个文件看作一个人，他对提 Bug 的人说了以下这段话：

“你好，我是Bug 登记员。

  1. 先自查：别问重复问题，别说方言（请用英语）。
  2. 给证据：别光嘴说，把报错的代码发给我，千万别发截图，我要能复制粘贴的文本！
  3. 报参数：把你电脑上相关软件的版本号告诉我，我看看是不是你环境装错了。

只有填完这些，我才收你的单子。”

一句话概括： 这是一个GitHub Issue 模版，目的是通过标准化提问格式，提高开发者修复 Bug 的效率。
