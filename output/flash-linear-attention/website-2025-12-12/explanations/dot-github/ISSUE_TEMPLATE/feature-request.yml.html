.github/ISSUE_TEMPLATE/feature-request.yml

完全没问题。这段代码看起来像天书，但实际上它就是一张“在线填空题”的设计图。

为了让你彻底搞懂，我们把这个过程想象成你正在为一家餐厅设计一张“顾客建议卡”。

我们可以把学习任务分成以下 4 个步骤（ToDo List）：

📝 学习任务清单 (To-Do List)

  • [ ] Task 01：搞懂大背景 —— 这文件到底是干嘛用的？
  • [ ] Task 02：设计卡片封面 —— 对应代码的前几行。
  • [ ] Task 03：设计填空题 —— 对应代码里的 body 部分。
  • [ ] Task 04：想象最终效果 —— GitHub 是怎么把这些代码变成界面的。

🚀 逐步讲解

✅ Task 01：搞懂大背景

场景： 你在 GitHub 上管理一个项目，很多人给你提建议（Feature Request）。 以前，大家随便写，有的写“我要个功能”，有的写“这不好用”，乱七八糟。 现在，你想立个规矩：“想提建议？请按我的格式填表！”

结论： 这个 .yml 文件，就是用来告诉 GitHub：“当有人点击‘提新建议’时，请给他弹出一个我设计好的表单。”

✅ Task 02：设计卡片封面

我们要看代码的最上面几行：

name: 🚀 Feature Request
description: Suggest an idea for this project.
title: "[Feature Request] "
labels: ["enhancement"]

翻译成人话：

  1. name (名字)：
    • 这是给你的表单起个名。
    • 效果：用户在菜单里会看到一个叫 "🚀 Feature Request" 的按钮。
  2. description (描述)：
    • 这是按钮下面的一行小字解释。
    • 效果：告诉用户“这是用来给项目提新点子的”。
  3. title (标题)：
    • 这是帮你自动填好的标题前缀。
    • 效果：用户打开表单，标题栏里已经自动写好了 [Feature Request]，省得他们乱写。
  4. labels (标签)：
    • 这是自动分类。
    • 效果：一旦用户提交，这个建议会自动贴上一个叫 enhancement (增强/改进) 的标签，方便你以后筛选。

✅ Task 03：设计填空题

这是文件的核心部分，也就是 body 下面的内容。这里定义了三个“填空框”。

第 1 个填空框：你的建议是什么？

  - type: textarea
    id: feature-request
    attributes:
      label: Feature Request
      description: Provide a clear and concise description of your feature proposal.
    validations:
      required: true
  • type: textarea：这是一个大文本框（能写很多字的那种）。
  • label：框上面的标题叫“Feature Request”（功能请求）。
  • description：提示语，“请清晰简洁地描述你的建议”。
  • required: true：必填项。不填不让提交。

第 2 个填空框：你为什么要提这个建议？

  - type: textarea
    id: motivation
    attributes:
      label: Motivation
      description: |
        Please explain the motivation... (此处省略长文)
    validations:
      required: true
  • label：框上面的标题叫“Motivation”（动机/原因）。
  • description：提示语，“请解释你为啥想要这个功能？是为了解决什么痛点吗？”
  • required: true：也是必填项。

第 3 个填空框：你能帮上什么忙吗？

  - type: textarea
    id: contribution
    attributes:
      label: Your Contribution
      description: |
        How can you contribute...
    validations:
      required: true
  • label：框上面的标题叫“Your Contribution”（你的贡献）。
  • description：提示语，“你能帮忙实现吗？比如你能不能直接写代码提交上来？”
  • required: true：也是必填项。

✅ Task 04：想象最终效果

当你把这个文件放进 GitHub 仓库后，神奇的事情发生了。

当用户点击 "New Issue"（新建问题）并选择 "Feature Request" 时，他们不会看到代码，而是会看到一个漂亮的网页表单，长得大概像这样：

标题： [Feature Request] (光标在这里闪烁)

Feature Request (必填) [ 这里是一个大大的输入框，让用户写建议 ] 小字提示：Provide a clear and concise description...

Motivation (必填) [ 这里是第二个输入框，让用户写原因 ] 小字提示：Please explain the motivation...

Your Contribution (必填) [ 这里是第三个输入框，问用户能不能帮忙 ] 小字提示：How can you contribute...

[ 提交按钮 ]

总结

这个文件不是给机器运行的程序逻辑，它是一份“配置清单”。

它只是在告诉 GitHub：“请帮我生成一个包含 3 个必填问题的问卷，专门用来收集大家的新点子。”
