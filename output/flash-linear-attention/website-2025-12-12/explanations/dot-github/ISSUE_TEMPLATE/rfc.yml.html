📋 学习任务清单 (To-Do List)

✅ Task 1：搞懂场景 —— 这文件是用在哪里的？

  • 背景知识：在 GitHub（程序员的代码仓库）上，有一个功能叫 Issue（议题）。通常大家用它来报 Bug 或者提建议。
  • 痛点：如果只给用户一个空白的大文本框，用户可能会乱写，比如只写一句“不好用”，开发者会很头疼。
  • 解决方案：开发者决定把“大白框”变成“填空题”。
  • 结论：你看到的这个 .yml 文件，就是用来告诉 GitHub 如何生成这个“填空题”表格的。

✅ Task 2：搞懂术语 —— 什么是 RFC？

  • 文件名：rfc.yml
  • 全称：Request for Comments（请求意见稿）。
  • 通俗解释：这是一种互联网圈的黑话。意思就是：“我有一个新功能的想法，但我还没开始做，我先写个草案，大家来投投票、提提意见。”
  • 结论：这个表格是专门给那些想提新功能建议的人准备的。

✅ Task 3：脑补画面 —— 代码变成了什么样？

现在我们开始看代码内容，请在脑海里想象一个网页表单：

  1. name: 📝 Request for Comments
    • 画面：当你想新建一个 Issue 时，GitHub 会弹出一个菜单让你选。这个选项的名字就叫“📝 Request for Comments”。
  2. description: ...
    • 画面：在这个选项下面有一行小字解释，写着“Propose a new feature...”（提议新功能并收集反馈）。
  3. title: "[RFC] "
    • 画面：当你点进去后，标题栏里已经自动帮你填好了前缀 [RFC]，你只需要在后面接着写你的标题。
  4. labels: ["enhancement"]
    • 画面：这个建议发出后，会自动打上一个“enhancement”（增强/改进）的标签，方便分类。

✅ Task 4：拆解核心 —— 表格里要填什么？

代码中最长的那段 body，就是具体的“填空题”题目：

  • 题目一：Proposal（提案）
    • 代码对应： ```yaml
      • type: textarea # 这是一个多行文本输入框 id: proposal attributes: label: Proposal # 题目叫“Proposal” description: ... # 提示语：请简述你的提议... validations: required: true # 【重点】这是必填项！不填不让提交。 ```
  • 题目二：Rationale（理由）
    • 代码对应： ```yaml
      • type: textarea # 这也是一个多行输入框 attributes: label: Rationale # 题目叫“Rationale” description: ... # 提示语：为什么你觉得这个功能很重要？ validations: required: false # 这是选填项，不写也可以。 ```

💡 总结

这一整段代码其实就是在对 GitHub 说：

“嗨，GitHub！如果有人想给我提新功能建议（RFC），请给他弹出一个表单。 标题自动帮他填个 [RFC]，标签打上 enhancement。 然后让他填两个空： 1. 你的建议是什么？（这是必填的） 2. 推荐理由是什么？（这是选填的）”
