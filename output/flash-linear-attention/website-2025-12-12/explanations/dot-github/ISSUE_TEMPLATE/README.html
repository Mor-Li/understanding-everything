.github/ISSUE_TEMPLATE

这是一个非常经典的 GitHub 项目配置。为了让你快速理解，我们把这个 .github/ISSUE_TEMPLATE 文件夹想象成一个“智能办事大厅”。

以下是通俗易懂的解读：

1. 这个文件夹（.github/ISSUE_TEMPLATE）主要负责什么？

角色：办事大厅的“规章制度”和“自动取号机”。

  • 以前（没有这个文件夹时）： 用户来反馈问题就像在大街上喊话。有人喊“救命”，有人喊“我要个苹果”，有人喊“今天天气不错”。开发者听得头大，因为信息太乱，不知道怎么处理。
  • 现在（有了这个文件夹）： 这里变成了一个规范的办事大厅。用户进来必须先填表。想报修？填红单子；想提建议？填绿单子。
  • 核心目的： 把用户的“胡言乱语”变成开发者能看懂的“标准文档”，提高沟通效率。

2. 各个直接文件分别是干什么的？

在这个“办事大厅”里，这 4 个文件就是 1 位门童和 3 种不同的表格：

  • 📄 config.yml —— 【智能门童 / 分诊台】

    • 作用：它是用户点击“新建问题”时看到的第一个界面。
    • 潜台词：它会先拦住你问：“你是来找茬（报Bug）的吗？如果只是想聊天或者问问题，请出门左转去 Discord 群组，别在这里排队。”
    • 功能：负责分流，把不该在这里提问的人指引到别的地方（如论坛、聊天室），把真正需要填表的人留下来。

  • 📄 bug.yml —— 【故障报修单】

    • 作用：专门用来接待“捉虫队”的。
    • 潜台词：“你说程序坏了？别光嘴说，请按格式填好：你的电脑型号、报错代码、复现步骤。不填完不许提交。”
    • 功能：强制用户提供修复 Bug 所需的技术细节。

  • 📄 feature-request.yml —— 【许愿卡 / 建议箱】

    • 作用：专门用来接待“产品经理型用户”的。
    • 潜台词：“你想要个新功能？行，请写清楚：你想要啥？为什么要这个？你能帮忙做吗？”
    • 功能：收集新功能的点子，并评估其必要性。

  • 📄 rfc.yml —— 【项目提案书】

    • 作用：专门用来接待“架构师型用户”的（RFC = Request for Comments）。
    • 潜台词：“你有一个可能会改变项目结构的大胆想法？别急着动工，先写份详细的草案（Proposal），大家开会投个票，论证一下合不合理。”
    • 功能：用于讨论比较重大的、伤筋动骨的改动，先行讨论，避免走弯路。

  • 通常作用：如果有子文件夹，它们通常是用来存放“附件”或“素材”的。比如存放一些图片、图标，或者更复杂的说明文档，用来辅助上面那些 yml 表格显示的更好看。但在你目前的结构里，全是直接的表格文件，非常扁平、高效。

4. 高层认知：一句话理解这部分代码

这部分代码不是“程序”，而是“流程”。

它不负责运行软件的功能，而是负责管理“人”。它通过预设的问卷调查（Templates），强迫提问者整理好思路再发言，从而保护开发者的发际线，防止他们被低质量的提问淹没。
