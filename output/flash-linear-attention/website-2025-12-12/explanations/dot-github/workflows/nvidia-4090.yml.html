.github/workflows/nvidia-4090.yml

这份文件实际上是一个 “自动化测试机器人的任务清单”。

在编程世界里，这叫 GitHub Actions 配置文件。它的作用是：每当有人修改了代码，GitHub 就会自动唤醒一台电脑，按照这个文件的指示，去测试代码有没有问题。

为了让你听懂，我们把这个文件想象成你（作为项目经理）写给一个测试员（机器人）的一张任务便签。

下面是这张便签翻译成人类语言的 To-Do List（任务清单）：

📋 机器人的执行清单 (To-Do List)

第一步：确认什么时候开工？(对应 on 部分)

  • [ ] 监听信号：
    • 如果有人发起 Pull Request（想合并代码进来），不管哪个分支，只要是新建、更新或重新打开的，马上开工。
    • 如果有人直接把代码 Push（推送）到了 main 主分支，马上开工。

第二步：管理排队情况 (对应 concurrency 部分)

  • [ ] 清理旧任务：
    • 如果同一个程序员手速很快，连续提交了两次代码，立刻取消上一在前一次提交上正在跑的测试。别浪费电，只测最新的那个版本。

第三步：准备工作环境 (对应 jobs 部分)

  • [ ] 确认任务名称：这次任务代号叫 Test 4090 (PyTorch 2.7)。
  • [ ] 最后一道检查：
    • 如果这个 Pull Request 已经被 closed（关闭/拒绝）了，那就别干了，直接下班。

第四步：把活儿“外包”出去 (对应 uses 部分)

这是这个文件最关键的地方！ - [ ] 调用通用模板： - 机器人，你不需要自己从头写测试脚本。请去隔壁文件夹拿那个叫 .github/workflows/reusable-ci-tests.yml 的通用测试手册。 - 意思是：具体的测试命令（比如“安装软件”、“运行代码”）都在那个文件里，这个文件只是负责传参数。

第五步：指定具体的配置参数 (对应 with 部分)

既然是调用通用模板，就得告诉模板具体的参数： - [ ] 指定电脑型号 (runner)：给我找一台装有 Nvidia 4090 显卡的机器来跑。 - [ ] 指定硬件类型 (gpu_type)：确认是 nvidia 的卡。 - [ ] 指定环境名 (conda_env_name)：Python 环境的名字叫 fla。 - [ ] 指定软件版本 (pytorch_version)：必须安装 PyTorch 2.7.0 版本。 - [ ] 显卡检查 (skip_gpu_check)：false -> 不要跳过显卡检查（必须确认显卡是好的）。 - [ ] 模型测试 (skip_models_tests)：true -> 跳过模型测试（可能为了省时间，这次只测基础功能，不测具体的模型跑分）。

总结一下

这个文件的核心思想是：

“嘿，GitHub！只要有人动了代码，就给我找一台 4090 显卡 的机器，装上 PyTorch 2.7，用我们写好的通用测试流程跑一遍。哦对了，如果是同一个人的重复提交，把旧的停掉省点资源。”

它就像是一个“点菜菜单”： * 菜谱是 reusable-ci-tests.yml (怎么做菜)。 * 这张纸是顾客的点单（我要用 4090 这口锅，炒 PyTorch 2.7 这道菜）。
