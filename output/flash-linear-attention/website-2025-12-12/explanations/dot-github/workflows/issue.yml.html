.github/workflows/issue.yml

没问题，这段代码其实就是给 GitHub 仓库雇佣了一个“自动清理机器人”。

它的核心逻辑是：定期扫描仓库里的 Issue（问题反馈），把很久没人理的 Issue 标记为“陈旧”，如果还没人理，就自动关掉。

为了让你好理解，我们把它想象成你给这个机器人列的一张 任务清单 (To-Do List)。我们可以按顺序把它拆解开：

🤖 机器人的工作手册 (To-Do List)

1. 设定上班时间 (When)

  • 代码对应： on: schedule: - cron: "0 0 * * 0"
  • 你的指令： “机器人，你不需要天天盯着。请在 每周日 的 午夜 0 点 醒来工作一次。”

2. 领取工牌和权限 (Permissions)

  • 代码对应： permissions: issues: write ...
  • 你的指令： “为了干活，我给你 修改 Issue 和 修改 Pull Request 的权限（这样你才能贴标签、发评论、关贴）。”

3. 设定“陈旧”的标准 (First Step)

  • 代码对应： days-before-issue-stale: 30
  • 你的指令： “如果一个 Issue 已经 30天 没有任何动静（没人回复、没更新），那它就是‘快要过期’的。”

4. 执行“陈旧”标记 (Action 1)

  • 代码对应：
    • stale-issue-label: "stale"
    • stale-issue-message: "This issue is stale..."
  • 你的指令： “对于上面那些 30 天没动静的 Issue，请做两件事：
    1. 给它贴上一个叫 stale (陈旧) 的标签。
    2. 在下面发一条评论：‘这个 Issue 已经 30 天没活动了，所以我标记它为陈旧。’”

5. 设定“VIP”豁免权 (Exceptions)

  • 代码对应： exempt-issue-labels: "enhancement"
  • 你的指令： “等一下！如果这个 Issue 身上已经贴了 enhancement (功能增强/建议) 的标签，不管它多久没动静，千万别动它，留着。”

6. 设定“关闭”的标准 (Second Step)

  • 代码对应： days-before-issue-close: 7
  • 你的指令： “在标记为‘陈旧’之后，再给它 7天 的宽限期。如果这 7 天内还是没人理它，那就准备动手。”

7. 执行“关闭”操作 (Action 2)

  • 代码对应： close-issue-message: "This issue was closed..."
  • 你的指令： “7天宽限期一过，直接把这个 Issue 关闭 (Close) 掉，并留言：‘因为标记陈旧后 7 天仍无活跃，现已关闭。’”

8. 关于代码提交的处理 (Pull Requests)

  • 代码对应：
    • days-before-pr-stale: -1
    • days-before-pr-close: -1
  • 你的指令： “注意，这个规则只针对 Issue。对于 Pull Requests (代码合并请求)，设置成 -1，意思是 完全不管，不要自动关闭别人的代码提交。”

总结：这篇文章（代码）的中心思想

这个文件的核心观点只有一句话： “为了保持项目整洁，我们要自动清理那些被遗忘的、不再活跃的讨论，把精力集中在新的或重要的问题上。”

整个流程图解： 1. 第0天： 有人提了个 Issue。 2. 第1-29天： 没人理。 3. 第30天： 机器人周日醒来，贴上 stale 标签，发警告评论。 4. 第30-36天： 依然没人理。 5. 第37天： 机器人周日醒来，直接 Close 掉这个 Issue。
