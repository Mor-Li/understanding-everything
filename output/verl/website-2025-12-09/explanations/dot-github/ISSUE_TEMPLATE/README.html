.github/ISSUE_TEMPLATE

好！我们继续用最接地气的方式来拆解这个文件夹。

你可以把 .github/ISSUE_TEMPLATE 这个文件夹想象成一个“办事大厅”（比如银行柜台或医院挂号处）。

下面是具体的比喻：

1. 这个文件夹主要负责什么功能？

核心功能： “分诊台”与“填表处”。

它的作用是立规矩。 当用户跑来找项目作者（医生/管理员）时，这个文件夹负责拦住用户，说：“别急着喊，先看你是来看病的（修 Bug），还是来提建议的（要新功能），然后按我的要求把这张表填好，否则不予接待。”

如果没有这个文件夹，用户就会像无头苍蝇一样乱喊，管理员会收到各种杂乱无章的信息。

2. 各个文件分别是干什么的？

在这个“办事大厅”里，这三个文件扮演了不同的角色：

  • 📄 bug-report.yml —— “急诊挂号单 / 报修单”

    • 场景：你的程序坏了，报错了。
    • 作用：这是一张专用的“病历表”。它强制你填写：哪里疼（报错信息）、吃了什么坏东西（运行环境）、怎么发作的（复现步骤）。
    • 目的：方便医生（开发者）快速确诊，修好代码。

  • 📄 feature-request.yml —— “意见箱 / 许愿单”

    • 场景：你觉得程序不够好，想要个新功能。
    • 作用：这是一张“建议书”。它强制你填写：你想要啥、为什么要这个、这能解决什么痛点。
    • 目的：防止用户异想天开，确保提的需求是有价值、有逻辑的。

  • 📄 config.yml —— “大厅经理 / 规则总开关”

    • 场景：控制整个大厅的运作方式。
    • 作用：它决定了是否允许“特殊通道”。比如里面写了 blank_issues_enabled: true，意思就是告诉保安：“如果有人不想填上面那两张表，只想拿张白纸随便写写，也放他进来吧。”

3. 高层认知：怎么快速理解这部分代码？

要理解这部分代码，你不需要懂编程逻辑，你需要懂“管理学”。

  • 不要把它看作代码：这部分内容不会让 verl 这个 AI 程序的算法跑得更快或更准。
  • 把它看作“沟通协议”：它是项目维护者为了保护自己的时间而设立的防火墙。
    • 它把“非结构化的抱怨”（比如：“你这破代码跑不起来！”）
    • 强行转化成“结构化的数据”（比如：“环境 Linux，Python 3.8，错误代码 404”）。

一句话总结： 这是项目组为了防止被乱七八糟的用户反馈淹没，而精心设计的一套“自动问卷调查系统”。
