.github/ISSUE_TEMPLATE/bug-report.yml

这个文件其实不是一篇“文章”或“代码逻辑”，而是一张“表格的设计图”。

它的作用是：当有人在 GitHub 上给这个项目（verl）提 Bug（报错/漏洞） 时，GitHub 会自动弹出一个表格让那个人填写。这个文件就是定义那个表格里有哪些必填项和选项。

为了让你更容易理解，我把你当作“想要报错的用户”，把这个文件转换成一份你需要完成的 To-Do List（任务清单）。

开发者通过这个文件，强制要求你在提问时完成以下步骤：

📋 提 Bug 流程任务清单 (Based on this file)

任务 1：确认你要干什么 (Metadata)

  • 文件里的逻辑：name: Bug Report, labels: [ "bug" ]
  • 你的任务：当你点击“New Issue”时，你会看到一个叫“Bug Report”的选项。一旦你提交，系统会自动给你的问题贴上一个“bug”的标签，方便管理员分类。

任务 2：接收感谢 (Markdown)

  • 文件里的逻辑：value: Thanks for taking the time...
  • 你的任务：阅读第一段话。开发者在感谢你花时间来反馈问题（客套话）。

任务 3：提供你的“体检报告” (System Info)

  • 文件里的逻辑：id: system-info, validations: required: true
  • 你的任务：（必填） 告诉开发者你的电脑环境。
    • 开发者甚至给了你作弊码：让你运行命令 python scripts/diagnose.py。
    • 你需要把运行出来的结果（比如 Python 版本、操作系统、verl 的版本）复制粘贴到这里。
    • 观点/目的：没有这些信息，开发者没法在你那边复现问题。

任务 4：勾选你的使用场景 (Information)

  • 文件里的逻辑：label: Information, type: checkboxes
  • 你的任务：做选择题。你是在用什么代码跑出了 Bug？
    • [ ] 官方提供的示例脚本 (The official example scripts)
    • [ ] 我自己修改过的脚本 (My own modified scripts)

任务 5：勾选你的任务类型 (Tasks)

  • 文件里的逻辑：label: Tasks, type: checkboxes
  • 你的任务：做选择题。你当时在跑什么任务？
    • [ ] 官方支持的任务（比如 GLUE/SQuAD 等）
    • [ ] 我自己的任务或数据集

任务 6：展示“犯罪现场” (Reproduction)

  • 文件里的逻辑：label: Reproduction, validations: required: true
  • 你的任务：（必填，最重要的一步） 教开发者如何让 Bug 再次出现。
    • 你需要提供代码片段（Code Sample）或者 Colab 链接。
    • 你需要列出步骤：第一步干啥，第二步干啥。
    • 警告：开发者明确要求不要用截图（Do not use screenshots），要直接贴代码文本，方便他们复制粘贴去测试。

任务 7：描述理想状态 (Expected behavior)

  • 文件里的逻辑：label: Expected behavior, validations: required: true
  • 你的任务：（必填） 告诉开发者，你觉得本来应该发生什么？
    • 比如：“程序应该输出结果，而不是直接崩溃报错。”

总结：这个文件想表达什么观点？

这个文件传达了项目维护者（Maintainers）的一个核心观点：

“如果你想让我们帮你修 Bug，请不要只扔下一句‘如果你程序坏了’就跑。你需要按标准格式，提供环境信息、复现步骤和代码，否则我们没法帮你。”

这是一个标准化管理的工具，用来提高沟通效率，防止用户提交无效的、信息不全的报错。
