CONTRIBUTING.md

完全没问题。这份文件其实就是一份“参与者指南”，告诉你如果你想给 Megatron-LM 这个项目贡献代码或提建议，应该遵守什么规矩。

你可以把它想象成你去别人家（NVIDIA）做客帮忙干活，主人家列了一张“干活须知”。

为了让你更容易理解，我把这些繁琐的文字转化成一个“任务清单 (To-Do List)”，你按照这个流程走就不会出错：

第一阶段：动手前的心理建设（了解背景）

  • [ ] 明白谁是老大：这个项目主要是 NVIDIA 内部团队在开发。虽然欢迎你帮忙，但你要知道，你的代码会先被拉到 NVIDIA 内部仓库，审核通过后再同步回 GitHub。
  • [ ] 判断任务大小：
    • 如果是修小 Bug：直接干，他们很欢迎。
    • 如果是大改动（改架构或大功能）：先别动手写代码！ 先发个帖（Issue）和他们讨论，否则容易白干。

第二阶段：如果你发现了问题（关于提 Issue）

如果你发现了 Bug 或者想提建议，发帖（Issue）时请勾选以下清单：

  • [ ] 选对模板：
    • 东西坏了？用 BUG 模板。
    • 以前能跑现在跑得慢或不准了？用 REGRESSION（回退）模板。
    • 想要新功能？用 ENHANCEMENT（增强）模板。
    • 单纯提问？不用模板，但要写清楚。
  • [ ] 专一原则：一个帖子只说一个问题。不要把三个 Bug 写在一个帖子里。
  • [ ] 提供证据：最好能提供“如何复现”的步骤，这样开发者能最快修好。
  • [ ] 注意语气：拼写语法要正确，语气要专业点。

第三阶段：如果你要提交代码（关于提 Pull Request / PR）

当你准备提交代码时，请检查你的代码是否符合以下“红线”：

✅ 必须做的 (Do): * [ ] 模仿风格：虽然没有强制的代码格式文档，但你要模仿现有的代码风格。 * [ ] 拆分提交：一个功能对应一次提交（Commit），不要把十个改动揉在一起。 * [ ] 基于 Master：确保你的代码是基于最新的 master 分支修改的 (Rebase)。 * [ ] 写好日志：Commit message（提交日志）要用英语，用祈使句（例如写 "Change X"，不要写 "Changed X"）。 * [ ] 检查拼写：代码、注释、日志里的英语单词别拼错了。

❌ 绝对禁止的 (Don't): * [ ] 不要违反开源协议。 * [ ] 不要“手欠”：只改你该改的地方。不要顺手去改别处的格式，这会让审核变难。 * [ ] 不要反复修改：设计好了再提交，不要在一个 PR 里反复推翻重来。 * [ ] 不要留垃圾：不要提交被注释掉的废弃代码。

第四阶段：提交之后（关于等待和催单）

提交了之后，你可能会焦虑“为什么没人理我”，请参考这个心理预期表：

  • [ ] 预期时间：通常 1周内 会有人看一眼或回复你。
  • [ ] 优先级排序：
    1. 最快处理：能复现的 Bug 和 Regression 修复。
    2. 看情况：新功能建议（如果他们觉得不符合项目方向，可能会直接关掉）。
    3. 最慢：求助帖（比如“这代码怎么跑啊？”），这种优先级最低。
  • [ ] 何时催单：如果 1周 没人理你，可以去 Ping 一下维护者。
  • [ ] 找谁催：目前的维护者是 @jaredcasper 和 @jon-barker。
  • [ ] 注意保鲜：如果你的 PR 超过 60天 没人动，会被机器人标记为“过期 (stale)”，记得及时去互动一下以免被关掉。

总结一下： 如果是小修小补，直接按规范提代码；如果是大事，先商量；提交后耐心等一周，不行再催。
